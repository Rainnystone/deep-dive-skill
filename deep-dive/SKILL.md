# Deep Dive 深度研究 Skill

## 核心定位

**普通搜索**找到信息
**Deep Dive**验证信息 + 分析因果 + 评估可信度

当用户需要：深度研究、交叉验证、探求原因、评估可信度、多源对比
**必须**调用此 skill，而非普通搜索。

---

## Phase 1: 广泛采集（必须并行执行）

### 搜索源配置（5+ 源并行）

#### 组 A: 主搜索链（Exa → Tavily → DDG fallback）
```
mcp__exa__web_search_exa
query: <主查询> + <扩展查询1> + <扩展查询2>
numResults: 10
includeRawContent: true

如果 Exa 失败（报错/无结果/额度满）:
→ mcp__tavily-search__tavily_search
query: <主查询>
max_results: 10
search_depth: advanced
include_raw_content: true

如果 Tavily 失败:
→ mcp__ddg-search__search
query: <主查询>
max_results: 10
```

#### 组 B: Jina 多源搜索（学术 + 新闻 + 博客）
```
mcp__jina__search_web
query: <主查询>
num: 10

mcp__jina__search_arxiv
query: <主查询>
num: 10

mcp__jina__search_jina_blog
query: <主查询>
num: 10
```

#### 组 C: Gemini 搜索（覆盖补充）
```
mcp__gemini-cli__ask-gemini
prompt: "使用 Google Search 搜索: <主查询>，返回至少10个最相关的结果，包括标题、URL和摘要"
```

#### 组 D: 扩展查询搜索（语义变体）
使用 mcp__jina__expand_query 生成 3-5 个扩展查询，对每个扩展查询执行：
```
mcp__exa__web_search_exa
query: <扩展查询>
numResults: 5
includeRawContent: true
```

## Phase 2: 内容获取（深度读取）

### 高价值 URL 筛选标准

从 Phase 1 的结果中，筛选以下 URL 进行深度读取：

1. **权威性**: 官方来源、知名媒体、学术机构
2. **相关性**: 与查询主题直接相关
3. **时效性**: 优先较新的内容（除非需要历史背景）
4. **独特性**: 包含其他来源没有的信息

### 并行读取

```
mcp__jina__parallel_read_url
urls: [精选的 URL 列表，最多 5 个]
withAllLinks: false
withAllImages: false
```

**注意**: 不要偷懒，尽可能读取完整内容而非仅依赖摘要。

## Phase 3: 深度分析（核心）

### 3.1 信息抽取

从所有来源中提取：
- **关键事实**: 具体数据、事件、声明
- **观点主张**: 各方立场、预测、评价
- **时间线**: 事件发生的先后顺序
- **参与方**: 涉及的个人、组织、利益相关者

### 3.2 交叉验证矩阵（增强版）

**目标**: 构建信息映射表，自动检测冲突。

#### Step 1: 信息点提取

从所有来源提取关键信息：
- **硬事实**: 数字、日期、名称、地点等客观信息
- **软信息**: 观点、评价、预测、立场
- **因果解释**: 原因分析、动机推测

#### Step 2: 语义对齐

识别同一事实的不同表述（避免误判冲突）：
```
语义等价示例:
- "价格上涨 20%" ≈ "价格增至 1.2 倍"
- "2024年3月" ≈ "去年春季"
- "公司裁员 500 人" ≈ "公司削减 15% 员工"

工具辅助:
- mcp__jina__deduplicate_strings 识别语义重复
- 手动核对数字/日期的数学等价性
```

#### Step 3: 冲突检测矩阵

| 检测类型 | 检测方法 | 处理方式 |
|---------|---------|---------|
| **数值冲突** | 数字不一致 >5% | ⚡ 标记冲突，分析原因 |
| **时间冲突** | 日期不一致 | ⚡ 标记冲突，检查时间线 |
| **事实冲突** | 相互矛盾的内容 | ⚡ 标记冲突，评估各方权威 |
| **观点分歧** | 立场/评价不同 | ⚠️ 标记分歧（非错误） |

#### Step 4: 冲突处理流程

对检测到的冲突执行：
```
1. 检查语义等价性 - 是否为表述差异而非真冲突？
2. 检查时间线 - 信息是否在不同时间点发布？（可能已更新）
3. 检查来源权威 - 各方来源的可信度如何？
4. 给出判断:
   - ✅ 可调和: 解释差异原因
   - ⚡ 真冲突: 标记哪方更可信，建议验证方向
```

#### 输出格式

| 信息点 | 来源A | 来源B | 来源C | 验证状态 |
|--------|-------|-------|-------|----------|
| 事实1 | ✅ | ✅ | ✅ | ✅ 3源验证 |
| 事实2 | ✅ | ✅ | - | ⚠️ 2源验证 |
| 矛盾1 | A说X | B说Y | - | ⚡ 冲突待处理 |

### 3.3 逻辑验证检查清单

对每个关键声明检查：
- [ ] **自洽性**: 数据内部是否一致？
- [ ] **因果合理性**: 原因和结果是否匹配？
- [ ] **时间线合理性**: 时间顺序是否合理？
- [ ] **来源独立性**: 多个来源是否独立？（避免同一原始来源的重复）
- [ ] **证据强度**: 有数据/官方确认 vs 传闻/推测

### 3.4 溯源分析（增强版）

**目标**: 追踪每个关键信息的最初来源，构建溯源链。

#### Step 1: 来源分类

对每个关键信息点识别来源层级：
- **Tier 1 一手来源**: 官方声明、原始数据、当事人直接陈述、学术论文原文
- **Tier 2 二手来源**: 引用 Tier 1 的报道、媒体转载、综述文章
- **Tier 3 三手来源**: 转载聚合、评论分析、社交媒体传播

#### Step 2: 链条追踪

对 Tier 2/3 来源，执行溯源追踪：
```
追踪方法:
1. 检查文章内的引用链接（"据...报道"、"...表示"）
2. 提取引用的原始来源名称/URL
3. 尝试访问原始链接验证内容一致性
4. 记录溯源链路径

使用工具:
- mcp__jina__read_url 读取原始来源验证
- 检查原文是否确实包含被引用的内容
```

#### Step 3: 溯源深度评分

| 溯源深度 | 评分加成 | 说明 |
|---------|---------|------|
| **完整溯源** | +15 | 成功追溯到一手来源，内容一致 |
| **部分溯源** | +10 | 追溯到二手来源，一手无法访问或不存在 |
| **无法溯源** | +0 | 无引用链、链接失效、引用模糊 |
| **溯源冲突** | -5 | 溯源结果与声明内容不符 |

#### Step 4: 溯源链输出格式

每个关键事实输出溯源链：
```
📌 事实: [具体内容]
溯源链: [一手URL/名称] → [二手A] → [二手B]
一手状态: ✅ 可访问且一致 | ⚠️ 可访问但有差异 | ❓ 无法访问
溯源深度得分: +15/+10/+0/-5
```

**溯源失败处理**:
- 记录无法溯源的事实为"待验证"
- 尝试通过其他搜索源寻找一手来源
- 在报告中明确标注溯源局限

### 3.5 因果深挖（向上探求）

对关键事件/现象，逐层分析：

**层级 1 - 直接原因**: 触发事件的具体因素
**层级 2 - 深层动机**: 各方行为背后的利益/目标
**层级 3 - 结构性因素**: 制度、文化、长期趋势
**层级 4 - 反事实思考**: 如果X没发生，会怎样？

### 3.6 可信度评分公式（v2）

每个信息点评分（0-115，含溯源加成）：
```
总分 = 来源权威性 (30) + 交叉验证 (30) + 证据强度 (25) + 时效性 (15) + 溯源深度 (15)

来源权威性:
- 一手官方/顶级学术: 30
- 权威媒体/知名机构: 25
- 普通媒体/自媒体: 15
- 匿名/未知: 5

交叉验证:
- ≥3 独立来源: 30
- 2 独立来源: 20
- 1 来源: 5

证据强度:
- 数据/官方文件/直接证据: 25
- 专家分析/引用来源: 15
- 推测/观点: 5

时效性:
- 最新/持续更新: 15
- 较新（<1年）: 10
- 较旧（>1年）: 5

溯源深度（见 3.4）:
- 完整溯源: +15
- 部分溯源: +10
- 无法溯源: +0
- 溯源冲突: -5

总分分级（调整为含溯源）:
- ≥95: ✅ 高置信度（完整溯源+多方验证）
- 70-94: ⚠️ 中置信度
- 45-69: ❓ 低置信度
- <45: ⚡ 待验证/存疑
```

## Phase 4: 输出约束（核心必含点）

**原则**: 不使用固定模板，Agent 根据研究结果自主组织输出形式。但必须包含以下核心点：

### 核心必含点清单

```
必须包含的内容点（顺序和详略由 Agent 自主决定）：

1. ✅ 执行摘要（TL;DR）- 3-5 句核心结论，包含最可信事实和最大不确定点
2. ✅ 可信度总览 - 高/中/低/存疑的统计分布
3. ✅ 关键发现 - 已验证的事实，每个事实带来源映射和评分
4. ✅ 矛盾/争议 - 信息冲突点分析（如有）
5. ✅ 溯源结果 - 一手来源追踪结果（见 Phase 3.4 溯源分析）
6. ✅ 因果分析 - 原因动机推断（如适用）
7. ✅ 局限性 - 研究缺口、未验证点、后续建议
8. ✅ 来源列表 - 按可信度排序的完整引用
```

### CLI 优化元素

Agent 在组织输出时应使用以下 CLI 友好元素：

```
## 状态标记系统
✅ 高置信（≥85分）| ⚠️ 中置信（60-84）| ❓ 低置信（<60）| ⚡ 存疑/冲突

## 紧凑表格格式
| 来源 | 可信度 | 核心信息 |
|------|--------|----------|

## 信息验证矩阵
| 事实 | 来源A | 来源B | 来源C | 验证状态 |
|------|-------|-------|-------|----------|
| X    | ✅    | ✅    | -     | ✅ 2源验证 |

## 溯源链可视化
📌 事实: [内容]
溯源链: [一手] → [二手A] → [二手B]
可信度: ⭐⭐⭐ (官方一手 + 2独立验证)
```

**禁止**:
- 大段无结构文字
- 过长无分割的章节
- 省略核心必含点中的任何一个

## 强制约束

### 不偷懒原则

1. **搜索**: 必须触发所有可用的搜索源，不能省略
2. **读取**: 高价值 URL 必须读取完整内容
3. **分析**: 每个分析维度必须执行，不能跳过
4. **输出**: 输出必须完整，不能因为长度而省略章节
5. **引用**: 每个关键结论必须有来源支撑

### 质量检查清单（v2）

输出前自检：
- [ ] **搜索覆盖**: 是否触发 ≥5 个独立搜索源？
- [ ] **深度读取**: 是否对高价值 URL 执行完整内容读取？
- [ ] **核心必含点**: 是否包含全部 8 个核心点？
  - [ ] 执行摘要（TL;DR）
  - [ ] 可信度总览
  - [ ] 关键发现（带来源映射）
  - [ ] 矛盾/争议分析
  - [ ] 溯源结果
  - [ ] 因果分析（如适用）
  - [ ] 局限性
  - [ ] 来源列表
- [ ] **溯源分析**: 是否对关键事实执行溯源追踪？
- [ ] **冲突检测**: 是否检测并处理信息矛盾？
- [ ] **可信度评分**: 是否使用 v2 公式（含溯源加成）？
- [ ] **CLI 格式**: 是否使用紧凑表格/状态标记而非大段文字？

## 错误处理

### 搜索源失败

如果某个搜索源失败：
1. 记录失败信息
2. 继续执行其他搜索源
3. 在报告中注明哪些源未成功
4. 评估对整体研究的影响

### 无结果处理

如果所有搜索源都无结果：
1. 尝试简化/泛化查询词
2. 尝试英文查询（如果原查询是中文）
3. 尝试相关关键词
4. 如果仍然无果，报告中如实说明搜索范围和建议

### 内容读取失败

如果 URL 读取失败：
1. 记录失败 URL
2. 尝试通过其他来源获取相同信息
3. 在来源列表中标注"内容未获取"
